Get Transaction Status and Details

Get Status of Transaction Request​

GET https://kernelserver.{env}.haloplus.io/consumer/QRCode/:reference

Asynchronously get Status of a transactions invoked through a link.

Path Parameters​

NameTypeDescription
env*StringThe backend environment [dev, qa, prod]

Query Parameters​

NameTypeDescription
:reference*StringThe id received when creating the intent transaction

Headers​

NameTypeDescription
x-api-key*StringThe API Key retrieved from the Merchant Portal
  • 200: OK Status of the transaction
{
    "qrCodeState": "ResponseReceivedByPhone",
    "transactionId": "693b47d8-e325-4808-8f86-67e388fd35ec",
    "merchantTransactionReference": "Transaction Test",
    "userId": "ebfbd120-76db-4a47-b040-f14ff7ee291f",
    "status": "AcknowledgedByPhone",
    "disposition": "Approved",
    "amount": "0.01",
    "currency": "ZAR",
    "type": "DebiCheck",
    "responseCode": "0",
    "authorisationCode": "3CA40B2A",
    "createdAt": "2023-03-01T10:35:02.585Z",
    "updatedAt": "2023-03-01T10:35:05.499Z",
    "paymentReference": "Example Ref";
}

Transaction Response Descriptions:​

FieldDescriptionType
qrCodeStateStatus of the deeplink createdString (64)
transactionIdID of the transaction when createdString (UUID v4)
merchantTransactionReferenceReference entered by the merchantString (255)
statusAt what point the transaction is in the transaction flow (ResponseReceivedFromPaymentProvider)String (64)
disposition

The outcome of the transaction (e.g. Approved)

There are 5 possible values for disposition​:

  • Approved​ - this means we got an approved message back from the processor
  • Declined​ - this means we received a decline from the processor
  • UnableToGoOnline​ - this means we couldn't connect to the processor. Essentially a decline, but it tells us the decline is due to a network error, and not a card issue
  • Indeterminate​ - this means we either received an error from the processor or no response. This means we are unable to determine the status of the transaction. Typically, we treat these as declines and then we attempt to void the transaction in the background, in the event that the processor got the message but we just didn't get the response
  • Voided​ - the transaction was reversed.
String (32)
amountValue of transaction (e.g 100.01)String
currencyCurrency of the amount (e.g. ZAR, GBP)String (3)
typeThe type of transaction (e.g. Tap, TT3, Secure card reader)String (32)
responseCodeThe ISO response code of the transactionNumber
createdAtISO date when the transaction was created
updatedAtISO date when the transaction was last updated
paymentReferenceReference of the transaction provided by customerString (255)

Get Tap on Own Transaction Status​

Get the status information of a tap on own transaction using the transaction's merchant reference.

GET https://kernelserver.{env}.haloplus.io/consumer/taponown/:merchantReference

Let’s take a closer look at the API request.

Path Parameters​

NameTypeDescription
env*StringThe backend environment [dev, qa, prod]

Query Parameters​

NameTypeDescription
:merchantReference*StringThe merchant reference provided to the Halo.SDK when starting a transaction.

Headers​

NameTypeDescription
Authorization Bearer*StringThe JWT received using the developer portal credentials (username and password)
note

To obtain this token you may need to make a POST call to https://authserver.{env}.haloplus.io/login and use { "username": "your-username", "password": "your-password" } as your json body.

  • 200: OK Status
  • 400: Bad Request
  • 500: Internal Error
{
    "id": "693b47d8-e325-4808-8f86-67e388fd35ec",
    "merchantReference": "My-Unique-Transaction-Reference",
    "disposition": "Approved",
    "errorCode": 0
}
{
    "errorCode": 122,
    "message": "E122: Transaction not found. It may have not gone online yet.",
    "httpStatusCode": 400
}
{
    "errorCode": 201,
    "message": "Error finding requested transaction",
    "httpStatusCode": 500
}

Get and Set Post Transaction Webhook​

Get the status information of a transaction if you are a Halo.SDK integrator or are performing intent (excluding the constant merchant id transaction flow) and deeplink transactions - otherwise known as app-to-app transactions.

1. App to App Transactions​

This API is currently unavailable and the webhook url has to be added manually. Please contact support@halodot.io for assistance.

2. Halo.SDK Integrator Transactions​

In order to receive the transaction status information through a webhook, you first have to register it through an API call.

POST https://appserver.{env}.haloplus.io/integrator/callbackUrl

Let’s take a closer look at the API request.

Path Parameters​

NameTypeDescription
env*StringThe backend environment [dev, qa, prod]

Request Body​

NameTypeDescription
callbackUrlStringThe fully qualified domain name to be called with the transaction status

Headers​

NameTypeDescription
Authorization Bearer*StringThe JWT received using the developer portal credentials (username and password)
x-api-key*StringThe API key retrieved from the developer portal
Json body Request
{
  "callbackUrl": "https://your-integrator-callback.top-level-domain",
}
  • 200: OK Status
  • 400: Bad Request
  • 401: Unauthorized
{}
{
    "errorCode": 104,
    "message": "E104: Invalid callback url",
    "httpStatusCode": 400
}
{
    "errorCode": 100,
    "message": "E100: No auth token",
    "httpStatusCode": 401
}

3. Webhook response​

The webhook will be called with a JSON body with following information:

Webhook JSON body response
{
  "id": "string",
  "merchantReference": "string",
  "disposition": "string",
  "errorCode": "number"
  "paymentProviderReference": "string",
}
FieldDescriptionType
idHalo backend transaction idString (UUID)
merchantReferenceUnique transaction reference provided by Halo.SDK integrating app.String
disposition

The outcome of the transaction (e.g. Approved)

There are 5 possible values for disposition​:

  • Approved​ - this means we got an approved message back from the processor
  • Declined​ - this means we received a decline from the processor
  • UnableToGoOnline​ - this means we couldn't connect to the processor. Essentially a decline, but it tells us the decline is due to a network error, and not a card issue
  • Indeterminate​ - this means we either received an error from the processor or no response. This means we are unable to determine the status of the transaction. Typically, we treat these as declines and then we attempt to void the transaction in the background, in the event that the processor got the message but we just didn't get the response
  • Voided​ - the transaction was reversed.
String (32)
errorCodeHalo backend error code (optional)Number
paymentProviderReferenceA reference received from the payment processor (optional)String

The callback url can be queried using an API call as well:
